# AGENTS.md: Submissions Pipeline

Constructs submission payloads from profile and program data, requests missing information via callback, and submits on the user's behalf. Pure function in, idempotent, retryable.

## Contract

See `.claude/docs/PIPELINE_CONTRACTS.md` for the full I/O spec.

Input: `{ profile_id, program_id, submission_id?, action: "prefill_only" | "submit", provided_data? }`
Output: one of `prefilled | needs_input | submitted | error`.

## Required components

- `deep_agent_langchain` node with MCP client wired to Ghost MCP server.
- `tool_tinyfish` in `application` mode: used only when `action=submit` and no missing fields remain.

## State machine

```
(new) 
  |
  v
[draft] --action=prefill_only--> [prefilled] or [awaiting_user_input]
  |
  |  (provided_data given)
  v
[awaiting_user_input] --fields resolved--> [ready]
  |
  |  action=submit
  v
[ready] --submit succeeds--> [submitted] --> [awaiting_program_response] (v2)
  |
  |  submit errors
  v
[error] (retryable or terminal)
```

Terminal states: `submitted`, `rejected`, `accepted`. `rejected` and `accepted` are set by v2 email ingestion (out of scope for v1).

## Behavior

### On first invocation for a (profile, program) pair

1. Read profile, program, and relevant `profile_narratives`.
2. Parse the program's application fields. For `apply_method=form`, derive the form schema from the scraped page content (stored in `program_pages`) or, if cheaper, re-inspect via Tinyfish in a read-only sub-step.
3. Generate `prefilled_fields` by mapping profile fields and narrative context to form fields.
4. Identify `missing_fields`: anything the form requires that the profile does not contain and cannot be reasonably inferred.
5. Write `submissions` row. Status = `prefilled` if nothing missing, `awaiting_user_input` if missing.
6. If `awaiting_user_input`, emit the `submission_needs_input` callback.

### On continuation invocation (submission_id present)

1. Read the existing `submissions` row.
2. Merge `provided_data` into the existing `provided_data`, then recompute `prefilled_fields`.
3. If `action=submit` and nothing is missing, transition to `ready`, call Tinyfish `application`, transition to `submitted` on success.
4. If still missing fields, remain in `awaiting_user_input` and emit another `submission_needs_input` callback with the updated (hopefully shorter) list.

### Idempotency

- If invoked with `action=submit` on an already-submitted row, return the existing `submitted` result. Do not resubmit.
- If invoked with `action=prefill_only` on a row in any state, re-run prefill but do not change terminal status fields. Never transition `submitted` backward.

## Two-stage prefill (important)

This pipeline owns **full** prefill generation. The short 3 to 5 sentence positioning summary shown in the match email is generated by the **scraping pipeline** and stored on `program_matches`. Do not regenerate it here. The full payload generated here lives on `submissions.prefilled_fields`.

## Writes to Ghost

- `submissions`: every invocation. Status transitions per the state machine.
- `program_matches`: update `status` to `applied` when a submission reaches `submitted`.

Never writes to `profiles`. If the submission surfaces gaps in the profile, the app layer sends the missing-info email which directs the user to the Profile page. Profile updates flow through the profile pipeline (conversational) or direct Wondergraph mutations (form).

## Callbacks

Emit one of:

- `submission_needs_input`: when status becomes or remains `awaiting_user_input`.
- `submission_submitted`: when status transitions to `submitted`.

Both per `.claude/docs/PIPELINE_CONTRACTS.md`. HMAC-signed.

## Does not send email

The app layer owns email composition and delivery. Both email types (match digest with submit deep link, and missing-info email) are composed in the app layer from Ghost state. Callbacks tell the app layer *when* to send.

## Tinyfish `application` mode

Invoke only when:
- `action=submit`
- Submission status is `ready` (no missing fields)
- Program's `apply_method=form` and `apply_url` is present

For programs with `apply_method=email` or `website_info_only`, `submit` may be a no-op that records a manual-apply instruction, or out-of-scope for v1. Pick one explicitly and document it in the implementation, do not silently half-implement.

## Latitude

- Model selection is yours.
- How to derive form schemas from scraped pages (LLM parsing vs structured HTML inspection via Tinyfish) is yours.
- Retry strategy on Tinyfish submission failures is yours. Transient network failures should be retryable; validation failures from the program's form should surface as `status: error, retryable: false`.

## Tests to include

- First-invocation prefill for a profile that covers all form fields. Verifies `prefilled` state, callback not fired (or fired as informational, implementor's call).
- First-invocation prefill with gaps. Verifies `awaiting_user_input`, callback fired with correct `missing_fields`.
- Continuation with partial `provided_data`. Verifies merge and remaining gaps.
- Continuation with complete `provided_data` and `action=submit`. Verifies Tinyfish call and `submitted` state.
- Duplicate `action=submit` on a `submitted` row. Verifies idempotency, no second Tinyfish call.
- Submission with `apply_method=email`. Verifies documented behavior (whichever you chose).
